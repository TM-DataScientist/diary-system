編集済みファイルに対して Linter を実行してください。

## 対象ファイルの決定

$ARGUMENTS が指定されている場合はそのファイルパスを対象にしてください。
指定がない場合は、以下の優先順位で対象ファイルを決定してください:

1. `git diff --name-only HEAD` で直近コミットから変更されたファイル
2. `git diff --name-only` でステージング済み＋未ステージのファイル
3. どちらも存在しない場合はユーザーにファイルパスを確認する

対象は `.ts` `.tsx` `.js` `.jsx` のいずれかの拡張子のみに絞ってください。

## 実行手順

### Step 1: 対象ファイルを確認してユーザーに表示

対象ファイルの一覧を表示し、Linter を実行する前に確認できるようにしてください。

### Step 2: Prettier チェック

```bash
npx prettier --check <対象ファイル...>
```

フォーマットが崩れているファイルがあれば一覧表示してください。

### Step 3: ESLint チェック

```bash
npx eslint <対象ファイル...>
```

エラー・警告の件数と内容を表示してください。

### Step 4: 結果のサマリーを表示

以下の形式でサマリーを出力してください:

```
## Lint 結果

対象ファイル: N 件

### Prettier
- ✅ フォーマット問題なし  /  ⚠️ N ファイルにフォーマット問題あり

### ESLint
- ✅ エラー・警告なし  /  ❌ エラー N 件、警告 N 件

---
[問題がある場合、自動修正するか確認する]
```

### Step 5: 自動修正の提案（問題がある場合）

Prettier や ESLint で問題が検出された場合は、ユーザーに自動修正を提案してください。
ユーザーが承認した場合のみ以下を実行してください:

```bash
npx prettier --write <対象ファイル...>
npx eslint --fix <対象ファイル...>
```

修正後は再度チェックを実行して、残存する問題がないか確認してください。
